# プレイテスト・フィードバックの処理

プレイの指摘は「テキストで説明を足す」のではなく、**根本原因を診断して構造で直す**。

## 1. 分類（まず原因のクラスを見極める）

指摘を次のどれかに当てる（複数可）。`harness/KNOWLEDGE.md` の対応原則を必ず参照：

| 症状の型 | よくある根本原因 | 参照原則 |
|---|---|---|
| 進めない／詰む | フラグ立て漏れ・中間セーブ・座標重なり・DOM遮蔽 | 原則 1, 6, 7, 8 |
| 何をしていいか分からない | ゴール／状態が世界の表示で語られていない | 原則 12, 15 |
| 押しても反応しない・意味不明 | 動詞の不一致・因果が不可視・UI遮蔽 | 原則 10, 11, 7 |
| やらされ感・ヒントが直接的 | 指示口調・見れば分かる状態の二重表記 | 原則 9（ヒントの3層） |
| 気持ちよくない・受け身 | 体感なし・演出が自動・納得感の欠如 | 原則 13, 16, 17 |
| 操作が分からない（ボタン等） | 効果を体感させず表記も操作説明どまり | 原則 13, 14 |

## 2. 診断（再現してから直す）
- 可能なら**スマホ実寸のスクリーンショット**を撮って目視で確認（判読性は自動検証できない）。
- 「テキストを足したくなったら負け」を疑う。**世界の表示・操作の体感・演出の因果**で解けないか先に考える。
- **フィードバックの翻訳層（原則0）**：指摘を字義通り実装する前に、「この問題を**コアファンタジーの
  語彙**で解けないか」を必ず一度通す。字義通りの実装が道具化（道具化テスト失敗）を進める場合は、
  コアファンタジー側の代替案を併記してユーザーに提示する。**テスターは摩擦を報告するが、
  ファンタジーの欠落は報告しない**——摩擦の解消がコアファンタジーを削る方向なら立ち止まる。

## 3. 修正
- `game-change` パイプラインに乗せる（設計→実装→ハーネス→監査→台帳）。
  設計時は `DECISIONS.md` の却下済み案に照らす——**指摘への対処案が過去に却下した案と同型なら、
  却下理由が今も生きているかを先に確認する**。分岐のある判断は `DECISIONS.md` へ追記。
- 直したら**同種の指摘が再発したらハーネスが検出できるか**を自問し、できないなら invariant / シナリオを足す。

## 4. 台帳
- `harness/KNOWLEDGE.md` に「症状→根本原因→対策」を1行追記。原則として一般化できるなら
  「設計原則」にも番号を振って足す。

## 5. 共有
- 修正版 `index.html` をユーザーに渡し（`SendUserFile`）、**何をどう直したか**と
  **ハーネス結果**を簡潔に報告。次に見てほしい点を一言添える。
- **定点質問**：次のプレイテストで「**◯◯している感じがしたか？（道具を使っている感じではなく）**」
  （◯◯＝このゲームのコアファンタジー）を必ず聞いてもらうよう一言添える。摩擦以外を測る唯一の手段。
