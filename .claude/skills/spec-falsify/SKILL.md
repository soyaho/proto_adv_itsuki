# 仕様反証ゲート（実装前に仕様を壊す）

AIとの合意形成で作った仕様は内部一貫性が高いぶん、**最初の思い込みを全工程に引き継ぐ**。
コードには独立監査（progress-auditor）があるのに、仕様には関門が無かった——これはその関門。
「作成者と評価者を兼ねさせない」を仕様レベルで実装する（原則47、`harness/KNOWLEDGE.md`）。

## いつ実行するか

- 仕様書・企画書をリポジトリに**新規に読み込んだとき**（テンプレートから立ち上げた
  プロトタイプに claude.ai 等のスパーリング産の仕様を持ち込んだとき、が典型）、
  **最初の実装より前に**
- 仕様を**実質的に改版したとき**（スタンプを外して再反証。誤字修正・体裁は対象外）
- セッション開始時に hook が「仕様反証リマインド」を出したとき（＝未反証の仕様書が
  リポジトリ直下にある）

## 手順

### 1. 対象の特定

リポジトリ直下の最新版の仕様書（CLAUDE.md「仕様書の鮮度ルール」——最新版だけが直下にある）。
引数でパスが指定されていればそれ。

### 2. 反証の実行（文脈を漏らさない）

`spec-falsifier` エージェントに**仕様書のファイルパスだけ**を渡して起動する。

- プロンプトに設計意図・経緯・この会話の要約・「ここは意図的」の類を**書かない**。
  漏らした瞬間、評価者が作成者バイアスに感染して別文脈の意味が消える
- 仕様の性質に応じて**追加の軸**が要る場合（取り返しのつかない選択・課金・対人要素・
  センシティブな題材など）は、軸の名前だけを足す（例:「炎上リスク軸も」）——
  懸念の中身・理由は書かない（それも文脈汚染）

### 3. トリアージ（本文脈で。鵜呑みにしない）

独立監査と同じ扱い——所見は鵜呑みにせず、1件ずつ次のいずれかに分類する：

1. **仕様を修正** — 同じ作業のうちに仕様書へ反映する
2. **判定待ちへ送る** — ハーネスで測れない懸念（手触り・退屈さ等）は `TODO_SYNC.md` の
   判定基準に足し、プレイテストで確かめる
3. **棄却** — 理由を明記する（「設計意図と違う」だけは棄却理由にならない。
   所見が事実誤認である根拠、またはコアファンタジー上その摩擦を受け入れる判断を書く）

分岐のある判断（仕様の方向を変えた・大きな所見を棄却した）は `DECISIONS.md` に積む。

### 4. スタンプと記録

- 仕様書の**先頭**に1行追記する：
  `反証: 済 YYYY-MM-DD（所見N件 → 修正a・判定待ちb・棄却c）`
  ——このスタンプが「実装に渡してよい」の印。hook はこの行の有無だけを見る
- トリアージの全文（棄却理由まで）は `TODO_SYNC.md` のタスク見出しに残す

## 注意

- ハーネスの緑と同じで、**反証済み＝面白いの証明ではない**（原則0）。反証は「最初の
  思い込み」の検出器であって品質保証ではない。コアファンタジーは引き続き設計工程と
  プレイテストで守る
- 反証の単位は**仕様の版**であって変更ではない。毎変更に反証をかけると所見が反復ノイズに
  なりゲートが形骸化する（→ 変更単位の関門は `/game-change` の設計工程が担う）
