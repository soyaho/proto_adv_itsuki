# 基盤修正の還流（このゲーム → テンプレート正本）

`framework/` と `.claude/` は各ゲームに vendored コピーされた**写し**で、正本は
テンプレートリポジトリ **`soyaho/html-game-template`**。
写しだけ直して終わると、次のゲームは直っていない基盤で立ち上がる。**還流はゲーム側の修正と同じセッションのうちに行う**
（後でまとめてやると「なぜその境界修正が必要だったか」の記憶が失われる）。

## 還流対象

1. `framework/` のコード変更（BaseDriver・ランナーの修正・拡張）
2. `harness/KNOWLEDGE.md` の「設計原則」に足した**一般化できる学び**
   （ゲーム固有のバグ台帳の行は還流しない。原則に昇格したものだけ）
3. `.claude/` のスキル・エージェント定義の改良のうち、ゲーム固有でないもの

## 手順

### 1. 還流対象の分離
- `git diff` で `framework/`・`.claude/` の変更と KNOWLEDGE.md の原則追記を列挙する。
- **ゲーム固有の知識（セレクタ・進行・座標・固有名）が混ざっていないか確認**。
  混ざっていたら、その部分はゲーム側の `harness/driver.js` や CLAUDE.md の
  「このゲーム固有の約束事」に移して基盤側は汎用のまま保つ
  （この分離作業こそが「境界の発見」＝テンプレートの育ち方）。

### 2. 正本リポジトリを同席させる
- `add_repo`（owner: `soyaho`, repo: `html-game-template`）でこのセッションに追加し、
  指示に従ってクローンする。
- アクセスが無い・拒否された場合は、ユーザーに GitHub 設定でのアクセス許可を
  依頼して**ここで中断**する（黙って諦めて写しだけ直した状態で終えない）。

### 3. 正本に適用
- `framework/`・`.claude/` の diff を正本の同じパスに適用する。
- 設計原則は正本の `harness/KNOWLEDGE.md`「設計原則」に追記する。
  **出自を添える**（どのゲームの・どんな事例から一般化したか）。

### 4. テンプレート自身のハーネスで検証（緑必須）
```bash
(cd framework && npm i) && (cd harness && npm i)   # 初回のみ
node harness/run.js                                # スターターの通しが green であること
```
- スターターの通しシナリオは **BaseDriver API の互換性テスト**を兼ねる。
- BaseDriver の破壊的変更でスターターが落ちたら、正本のスターター側
  `harness/driver.js` も同じコミットで追従させ、**破壊的変更である旨を
  コミットメッセージに明記**する（既存ゲームの写しを更新するとき要注意、の印）。

### 5. コミット＆プッシュ（正本へ）
- メッセージに「**どのゲームで見つけた、どんな境界修正か**」とハーネス結果を書く。

### 6. 整合の確認
- 還流後、ゲーム側の `framework/` と正本の `framework/` が一致していることを
  diff で確認する（ゲーム側だけ新しい／正本だけ新しい状態を作らない）。
