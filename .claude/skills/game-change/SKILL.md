# ゲーム変更パイプライン

`index.html` に触る変更は、必ずこの順で行う。**一つでも飛ばさない。**
前提知識として CLAUDE.md の「このゲームの構造」（状態モデル・入力の動詞・進行の節目・UIモード）を先に読む。

## 1. 設計
- **変更が仕様書・企画書に由来するなら、その仕様書が反証スタンプ（先頭の `反証:` 行）を
  持つことを先に確認する。** 無ければこの工程に入る前に `/spec-falsify` を実行する
  （CLAUDE.md「仕様反証ゲート」・原則47）。
- **最初に原則0（コアファンタジー・ゲート）**：ギミックに触れるなら「テーマを剥がした汎用の
  道具に差し替えても遊びが変わらないか？」（道具化テスト）を問い、判定を設計メモに書く。
  変わらないならコアファンタジーの動詞で言い換えてから先へ進む。
- 変更内容と、影響する **進行フラグ・入力対象・UIモード**（CLAUDE.md「このゲームの構造」で
  定義したもの）を列挙する。
- 該当する設計原則（`harness/KNOWLEDGE.md` の「設計原則」）に照らす。
  特に UX 変更なら：ヒントの3層／動詞の文法／因果を演出で見せる／入力ターゲットを曖昧にしない。
- **`DECISIONS.md` の却下済み案に照らす**——却下済みの案を知らずに再提案しない。覆すなら
  却下理由を覆す根拠を添えて新しい決定として積む。**分岐のある判断をしたら、決定・理由・
  却下した代替案を同じコミットで `DECISIONS.md` に追記する**（後でまとめて書かない）。

## 2. 実装
- 規模が大きければ `implementer` サブエージェントに実装を委任し、本文脈はレビューに回る。
- 約束事（`CLAUDE.md`「実装上の約束事」——普遍の節と、このゲーム固有の節の両方）を守る。
  特に：新しい進行フラグはセーブ対象に足し旧セーブで falsy 安全に、進行フラグは操作の瞬間に確定、
  入力ターゲットの判定を重ねない・遮蔽しない。
- 基盤（`framework/`）に直しが必要なら触ってよい。ただし**ゲーム固有の知識（セレクタ・
  進行・座標）を持ち込まず**、触ったら同じセッションのうちに `/framework-backflow` を
  実行して正本へ還流させる。

## 3. ハーネス実行（緑必須）
```bash
node harness/run.js
```
- desktop / mobile縦 / mobile横 の全シナリオが green であること。
- **新しい遊び・進行を足したら、同じコミットで `harness/scenarios/` を拡張する**
  （そのイベントに実操作で到達するステップを通しシナリオに追加）。
- 実入力主義：`page.evaluate` は読み取り専用。状態書き込み・内部関数呼び出しは禁止。

## 4. 独立監査
- 進行フラグ／入力処理／インタラクション対象の配置／セーブ／モード遷移に触れたら、
  **`progress-auditor` エージェント**を別文脈で回す（進行不能監査）。
- 所見は鵜呑みにせず、**ハーネスのシナリオで再現できたものだけ**をバグとして扱う。

## 5. ノウハウ反映
- 新種のバグを見つけたら `harness/KNOWLEDGE.md` のバグ台帳に1行追記
  （症状→根本原因→対策）。**書くだけで終わらせず**、可能なら driver の invariant か
  シナリオのアサーションに落とす。
- ゲームの構造（状態モデル・動詞・節目・モード）が変わったら、CLAUDE.md の
  「このゲームの構造」「このゲーム固有の約束事」も同じコミットで更新する。

## コミット
- コミットメッセージに**ハーネス結果（プロファイル別 pass/fail の要約）を一行**入れる。
- 指定された作業ブランチにコミット・プッシュ。
