# 設計判断ログ（DECISIONS）

このゲームの**分岐があった設計判断**を記録する台帳。運用は CLAUDE.md「設計判断ログ」の節。
仕様書には結論しか残らない——「なぜそうしたか・何を捨てたか」の正本はここ。
設計に入る前に必ず参照し、**却下済みの案を知らずに再提案・再実装しない**ための器。

## 書き方

- **1決定 = 1ブロック**。新しい決定を上に積む
- ID は `D1, D2, …` の恒久連番（振り直さない。コミット・他台帳から参照される）
- 見出しの状態は「有効」か「superseded by D#」。覆すときは旧ブロックを**書き換えず**、
  新しい決定として積み、旧見出しの状態だけ書き換える（却下理由を覆した根拠を新側に書く）
- 書くのは分岐があった判断だけ。迷わなかった実装詳細・命名は書かない
- **各決定に「出所と確信度」を1行付ける（D4以降は必須）**——出所＝その判断（特に却下理由）を
  何で決めたか（AI対話 / 実プレイ観察 / 実測〔ハーネス・数理〕/ 外部事実）、確信度＝高・中・低。
  却下案ごとに出所が違うときは各行に併記する。**出所がAI対話で確信度が高くない決定は
  「未検証判断」**——プレイテストを組むとき（TODO_SYNC の判定待ち）に観察対象として
  判定基準へ拾う（運用は D5。欄の無い D4 以降のブロックは SessionStart hook が警告する）
- claude.ai 等の設計スパーリングから持ち込むときは、仕様の結論だけでなく
  **却下した代替案とその理由**をここへ移す
- 肥大化したら superseded の古いブロックから `_archive/` へ（有効な決定は残し続ける）

## D13: フタの開け直しは「フタを置き場に戻す」＝D8の文法で行う（独立監査所見1） — 有効

- 日付: 2026-07-16
- 出所と確信度: 実測（progress-auditor の座標検証: lid_on↔suitou が中心間52px＜半径和62px）・
  確信度 高
- 決定: 閉まったフタを開ける操作は、専用ターゲット（lid_on）ではなく「フタの置き場
  （ゴースト表示）をタップ→フタが置き場に戻る＝開く」にする。置き場にはフタが無い間ずっと
  ゴーストを描く（可視ヒント）。ハーネス driver に「判定円の重なりは半径和−8pxまで」の
  下限チェックを追加（このクラスの恒久検出）
- 理由: フタは物理的に水筒の上に乗るため、別ターゲット方式はどう配置しても判定が重なるか
  見た目とタップ位置が乖離する。「取った場所に戻す＝キャンセル」（D8）の文法をフタにも
  適用すれば、追加の文法なしで開け直しが成立し、ゴーストが D12 と同じ言語で場所を語る
- 却下した代替案:
  - lid_on ターゲットを水筒の上に置く（当初実装） — 中心間52pxで判定重なり。フタ狙いの
    タップが水筒ピックアップに化ける（6歳の誤爆点）
  - suitou の判定半径を縮めて分離 — 当たり判定最小60×60px（仕様§3.1）に違反する
  - lid_on を離れた位置に置く — 見た目（フタは水筒の上）とタップ位置が乖離し、
    「見えるものをタップする」原則が崩れる

## D12: ゴール提示は「世界に常設するゴースト」で行う（所見1） — 有効

- 日付: 2026-07-16
- 出所と確信度: AI対話（spec-falsify 所見1のトリアージ）・確信度 中（**未検証判断**——
  「考える」を奪っていないかはプレイテストで観察。TODO_SYNC 判定基準へ）
- 決定: ゴール条件（文字表示禁止）の代替は、各席のランチョンマットに置き場所のうすい形
  （ゴースト）を常設＋カバンの中に持ち物のうすい形を常設。開始時のママ吹き出しは
  雰囲気の一言のみで、手順は語らない
- 理由: 反証所見1（ゴール提示手段の未定義＝遊び始められない）は入口を塞ぐ致命。
  ゴーストは「完成形の絵がゴールを語る」（原則5）を最も文字数ゼロで実装でき、
  現実のランチョンマット・持ち物ポケットという実在物のメタファに乗る
- 却下した代替案:
  - シーン開始時に完成形の絵を1〜2秒見せるデモカット — 受け身（KNOWLEDGE 原則13・16）で、
    6歳は見ていない可能性が高い。常設なら迷った瞬間にいつでも参照できる
  - ママの吹き出しにミニ完成図 — 吹き出しは時限表示で消える。消えた後に迷子が再発する
  - 誘導矢印 — 仕様§11が明示的に禁止（「考える」が「当てる」に劣化する）

## D11: おしぼり・コップはまとめ持ち可、箸だけ1膳ずつ（所見6） — 有効

- 日付: 2026-07-16
- 出所と確信度: AI対話（所見6のトリアージ）・確信度 中（**未検証判断**——退屈さの実際は
  プレイテストで観察。TODO_SYNC 判定基準へ）
- 決定: おしぼりの山・コップの重ねは束のまま持ち、置き先タップごとに1つずつ置く。
  箸は1膳ずつ運ぶ。みずさし1回の往復で3杯注げる
- 理由: 素の仕様は最低9往復の純作業で、6歳が3往復目で飽きる（所見6）。一方「数を考える」
  ポイントは箸に集約されているので、箸だけ1膳ずつ残せば考えるポイントは失われない。
  束持ち・重ね持ちはそれ自体が現実の朝の知恵（まとめて運ぶ）のリハーサルになる
- 却下した代替案:
  - 全部1個ずつ（仕様のまま） — 9往復×7秒の歩行鑑賞は「おなじことばっかり」で離脱
  - 箸もまとめ持ち — 「家族は3人だから3膳」を数える瞬間が消え、考えるポイントの1つを失う

## D10: 描画は Canvas（仕様の SVG 推奨から変更） — 有効

- 日付: 2026-07-16
- 出所と確信度: 実測（テンプレートのハーネス基盤の実装）＋AI対話・確信度 高
- 決定: 描画は SVG ではなく Canvas＋`tapTargets()` 公開で実装する
- 理由: テンプレートのハーネス基盤（BaseDriver の論理座標タップ・座標分離 invariant・
  遮蔽チェック）が「canvas＋tapTargets()」を前提に実証済み。SVG にすると driver の
  タップ変換・invariant を作り直すことになり、配線実証済みの検証資産を捨てる。
  キャラアニメも固定タイムステップ＋timescale 注入（原則39）と Canvas の相性がよい
- 却下した代替案:
  - SVG（仕様推奨） — オブジェクト単位のタップ判定は素直だが、ハーネス資産の作り直しと
    「アニメは CSS/WAAPI」への分散（タイムスケール加速が効かない）が高くつく

## D9: 日焼け止めは「鏡」をインタラクション面にする（所見8） — 有効

- 日付: 2026-07-16
- 出所と確信度: AI対話（所見8のトリアージ）・確信度 中
- 決定: チューブを持って鏡をタップ→いつきくんが鏡の前に立つと鏡が顔を大きく映す→
  「鏡の中の顔」の3ヶ所をタップして塗る。タップ対象は常に世界のオブジェクト（鏡）
- 理由: 素の仕様「顔をタップ」はゲーム全体で教えた「タップ＝そこへ歩け」の文法をこのタスク
  だけ破る（所見8）。鏡を対象にすれば文法が一貫し、塗り残しの視認（大きく映る）も同時に解決
- 却下した代替案:
  - いつきくん本人をタップして塗る — 自分自身へ歩けず文法が折れる。誤タップ（移動指示との
    衝突）も避けられない
  - 塗りは自動（鏡の前に立てば完了） — 「塗る」型（3.3）の操作の実感が消える。
    受け入れ基準5「自分の操作が結果につながった」に反する

## D8: 持ち状態の文法＝「置けるのは置き先だけ・取った場所で戻せる・他は反応のみ」（所見3） — 有効

- 日付: 2026-07-16
- 出所と確信度: AI対話（所見3のトリアージ）・確信度 中
- 決定: 持っている間も世界の反応（原則3）は変わらない。置けるのは受け側だけ。
  取った場所を再タップで戻す（キャンセル）。それ以外では持ち物は手から離れない
- 理由: 未定義のままだと「どこにも置けない／変な場所に置かれる」が全シーンで再発（所見3）。
  「手が塞がって詰む」状態を構造的に排除しつつ、誤操作で持ち物が散乱しない
- 却下した代替案:
  - どこでも置ける（床置き自由） — 6歳の誤タップで持ち物が散乱し、拾い直し作業が発生。
    片付けゲームではないので自由度の価値がない
  - 持ち直し不可（置くまで拘束） — 3膳目の箸（1-B）のように「戻す」が正解の局面が
    仕様に存在するため、キャンセル動詞は必須

## D7: 受理は即時（波紋＋歩き出し）・結果は接触時、進行フラグは接触の瞬間に同期確定（所見4） — 有効

- 日付: 2026-07-16
- 出所と確信度: AI対話（所見4のトリアージ＋KNOWLEDGE 原則33）・確信度 中
  （移動待ちのテンポは**未検証判断**——プレイテストで観察。TODO_SYNC 判定基準へ）
- 決定: タップの瞬間にタップ地点へ受理マーク（波紋）＋いつきくんが即座に歩き出す。
  オブジェクトのリアクション・進行フラグの確定は、いつきくんが接触した瞬間に**同期的に**行う
  （接触ハンドラ内で確定。タイマー・演出コールバックに逃がさない）
- 理由: 即時リアクションにすると原則2「体が挟まる」が崩れ、到着時のみにすると受理が
  帰属不明になる（KNOWLEDGE 原則33）。受理と結果を分離すれば両立する。
  「操作の瞬間に確定」（KNOWLEDGE 原則6）はこのゲームでは「接触の瞬間」と読み替える——
  タップは指示であり操作の完了は接触だから。セーブ不使用（§7）なので中間セーブ窓の詰みは
  構造的に存在しない
- 却下した代替案:
  - タップ即時に世界が反応 — 「指と結果の間に体が挟まる」（原則2）が崩れ、持つ/置くの
    文法学習と衝突する（所見4）
  - 到着時のみ反応（受理表示なし） — 受理が見えない操作は帰属不明（原則33）。
    連打・目的地上書きと相まって「押したのに効かない」に見える

## D6: 世界ルール「ふたのない みずは、はこぶと こぼれる」＋みずさし導入（所見2・7） — 有効

- 日付: 2026-07-16
- 出所と確信度: AI対話（所見2・7のトリアージ＋KNOWLEDGE 原則19・49「理は1本」「ルールは冒頭宣言」）・
  確信度 中（**未検証判断**——6歳がこのルールを1回のこぼしで学習できるかはプレイテストで観察）
- 決定: 水の物理は「開いた容器＋水＋歩行＝こぼれる（水たまり化・拭ける）／閉じた容器は安全」の
  1本に固定。シーン1に「フタつきのみずさし」を追加し、正解経路（コップを席に置く→みずさしで
  運んで注ぐ）を成立させる。コップ直持ちで蛇口から入れる経路も残す（こぼして学ぶ発見経路）。
  こぼれは容器が空になるだけで他の進行を巻き戻さない
- 理由: 素の仕様は §3.3「容器を持って対象タップ＝注ぐ」と §4.2「先にテーブルに置く→蛇口タップで
  （離れたコップに）水が注がれる」が自己矛盾しており（所見2）、どちらを実装しても片方と食い違う。
  みずさし（フタ＝閉じた容器）なら §3.3 の型のまま矛盾なく解け、シーン2の「フタを閉めてから
  カバンへ」と同じ理で説明できる（原則19）。「こぼれ＝やり直し」は罰体験（所見7）なので、
  損失をその容器の水だけに限定し、拭く動詞を楽しい系に振る
- 却下した代替案:
  - §4.2 を字義通り実装（テーブルのコップに蛇口から瞬間移動で注水） — 世界の物理が
    その場しのぎになり、「なんでこれで正解なのか分からない」を生む（原則49）
  - 蛇口の下でしかコップに注げない・持ち歩き禁止 — 「こぼす」という発見経路と
    「拭くとまた楽しい」（仕様の意図）が丸ごと消える
  - こぼれたらタスクやり直し（仕様の字面） — 6歳には罰として体験される（所見7）。
    原則1「失敗演出を出さない」と実質矛盾

- 日付: 2026-07-13
- 出所と確信度: AI対話（ギャップ3の指摘＋既存台帳の構造分析）・確信度 高
  （追記式の1行メタデータで、失うものが無い）
- 決定: 各決定ブロックに「出所と確信度」を1行付ける（出所: AI対話 / 実プレイ観察 / 実測 /

  必須とし、欄の無い D4 以降のブロックは SessionStart hook が警告する。プレイテスト
  （TODO_SYNC の判定待ち）を組むときは、**出所がAI対話で確信度が高くない決定を観察対象として
  判定基準へ拾う**——「未検証のままAI合意で確定した判断」を現実の観察へ回す循環を作る
- 理由: 却下理由は再提案を封じる効力を持つのに、AI対話だけで決めた却下と実プレイで検証済みの
  却下が同じ権威で並んでいた。出所と確信度が付けば、覆すときのハードルが証拠の強さに比例し、
  未検証判断が「観察待ちリスト」として機械的に浮き上がる。追加コストは1行で、
  既存の運用（決定はその作業のうちに書く）にそのまま乗る
- 却下した代替案:
  - D1〜D3 への遡及タグ付け — 「旧ブロックを書き換えない」原則に反する上、記憶からの
    出所復元は捏造になる。覆す必要が出た決定から新ブロックで積み直せば足りる
  - 独立した「証拠台帳」の新設 — 台帳が増えるほど参照されなくなる（D1 の統廃合と同根）。
    既存の決定ブロックへの1行追記で同じ情報が乗る
  - 確信度の数値化（%表記） — 根拠のない精度の錯覚を生む。「観察対象を拾う」用途には
    3段階で十分

## D4: 仕様反証ゲート（/spec-falsify）を正規工程として追加する — 有効

- 日付: 2026-07-13
- 出所と確信度: AI対話（外部の運用助言＋テンプレート構造の分析）・確信度 中
  （ゲートが形骸化しないか・所見の質が実用に足るかは、最初の実プロジェクトでの運用で観察する）
- 決定: 仕様書を実装に渡す前に、作成者と文脈を共有しない別文脈のエージェント
  （`spec-falsifier`・仕様書のパスだけを渡す）に「実プレイヤーの離脱理由を操作・退屈・
  意味不明の3軸で挙げさせる」反証工程を必須化する（原則47）。発動は3層で自動化する：
  ①CLAUDE.md「仕様反証ゲート」（スタンプの無い仕様書から実装しない規約）
  ②SessionStart hook（直下の台帳以外の .md に `反証:` スタンプが無ければ警告）
  ③`/game-change` 工程1（設計前のスタンプ確認）。所見は独立監査と同じく鵜呑みにせず、
  修正／判定待ち送り／棄却＋理由にトリアージして仕様書スタンプと TODO_SYNC に記録する
- 理由: このテンプレートは「作成者と評価者を兼ねさせない」をコード（progress-auditor）には
  持つが、仕様が実装に渡る関門が無かった。主な仕様の入口は claude.ai でのAIスパーリング産で
  （CLAUDE.md がその持ち込みを正規手順にしている）、AIと合意形成した仕様は内部一貫性が
  高いぶん最初の思い込みを全工程に引き継ぐ。原則0の「設計工程で守る」の設計工程自体が
  自己点検のみだった。既存機構（サブエージェント＝別文脈・スキル・鮮度ルールのヘッダ行・
  既存 hook）だけで実装でき、仕様が無ければ何も起きない（需要駆動。D3 と同根）
- 却下した代替案:
  - 変更ごとに反証を必須化（/game-change の工程0に組み込む）— 反証の単位は「仕様の版」で
    あって「変更」ではない。毎変更では所見が反復ノイズになりゲートが形骸化する
    （オオカミ少年化。D3 の定期CI却下と同じ症状）
  - 同一文脈での自問チェックリスト — 作成者バイアスを共有した評価は反証にならない。
    これがこのギャップの指摘の核心（安価だが目的を満たさない）
  - hook で「仕様を読み込んだ」を直接検知して強制起動 — Claude Code の hook は読み込みを
    意味論的に検知できない。ファイル規約（反証スタンプ）＋セッション開始時の走査が
    機械検知できる最大で、これで十分（スタンプの無い仕様書は毎セッション警告され続ける）
  - ギャップ2（実プレイヤーへ渡す閾値）も同時に自動化 — 現実の人間の観察は定義上
    自動化できない。受け渡しの器は TODO_SYNC の「判定待ち＋判定基準」として既に存在する。
    自動化ではなく運用（観察指標をKPIにする）の問題

## D3: 鮮度維持は定期CIではなく「使用時キャッチアップ」で行う — 有効

    反する。利用頻度が上がり「常に誰かが見ている」状態になったら再検討してよい
