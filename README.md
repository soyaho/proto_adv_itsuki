# html-game-template

どんなブラウザゲームでも立ち上げられる、**Claude で開発するためのハーネスだけが最初から
整っている**根幹テンプレート。ジャンルの規約（アドベンチャー・パズル・カード…）は持たない。
持っているのは「実入力テストハーネス・開発ワークフロー・設計原則の台帳・サブエージェント」
という**開発の骨格**だけ。

> ジャンル規約を外して抽出した親テンプレート。ジャンル特化テンプレートはここから派生させる。
> **正本はこのリポジトリ（`soyaho/html-game-template`）**——派生ゲームからの還流（`/framework-backflow`）はここに集まる。
> ワークフロー・設計原則の正本もここ（上位テンプレートへの多段還流は廃止。`DECISIONS.md` D1）。

## 入っているもの

| パス | 中身 |
|---|---|
| `index.html` | **配線確認用スターター「三つの灯」**（1画面・依存ゼロ・動く）。ゲームというより、実入力・セーブ復帰・invariant の配線が正しいことの証明＋driver/シナリオの書き方の見本。最初の実装で丸ごと置き換える |
| `framework/` | **real-input-harness** — ゲーム非依存のテスト基盤（Playwright）。汎用ランナー＋ BaseDriver（実タップ変換・step記録・遮蔽/座標重なりの invariant） |
| `harness/` | このゲーム固有のテスト。スターター用の `driver.js`（BaseDriver 拡張）と通しシナリオ、**設計原則＋バグ台帳** `KNOWLEDGE.md` |
| `CLAUDE.md` | 開発規約。変更パイプライン（設計→実装→ハーネス緑→独立監査→台帳）を強制し、**ゲーム固有の規約を育てるスロット**を持つ |
| `TODO_SYNC.md` | タスクの完了記録＋プレイテスト判定待ちの器。ハーネスの緑と「人にしか判定できないもの」を分離して見せる |
| `DECISIONS.md` | 設計判断ログ。1決定＝1ブロックで「決定・理由・**却下した代替案と理由**」を残し、却下済み案の再提案を防ぐ。設計スパーリング（claude.ai 等）の結論の受け皿 |
| `.claude/` | Claude 用ワークフロー一式：`/game-change` `/playtest-fix` `/run-harness` `/framework-backflow` `/template-catchup` スキル、`implementer` / `advisor` / `progress-auditor` / `harness-runner` サブエージェント、依存自動導入の SessionStart フック |

## 適用範囲 — どんなゲームのベースになるか

**対象はブラウザで動くゲーム全般**（canvas でも DOM でも、ジャンル不問）。ただし層ごとに
効き方が違うので、立ち上げるゲームに合わせて取捨する：

| 層 | 汎用度 |
|---|---|
| 開発ワークフロー（CLAUDE.md・`.claude/`・台帳） | ゲームであれば何にでも効く |
| ランナー（`framework/runner.js`） | ブラウザゲームなら何にでも使える（プロファイル×シナリオ実行） |
| `BaseDriver`（`framework/driver.js`） | 起動・step記録・読み取り専用 `read()` は汎用。**論理座標タップと遮蔽系 invariant は「2D・単一canvas」向けのオプトイン**——DOM 主体のゲームは page.click / locator だけでも成立する |
| スターター＋ゲーム固有 driver／シナリオ | 見本。ゲームごとに丸ごと書き換える |

ハーネスの根本思想は「**実入力で操作 → 状態をポーリングで待つ**」。ここから来る得手不得手：

- **ターン制・プレイヤーを待ってくれるゲーム** — フルに効く（到達可能性の証明・遮蔽検出・セーブ復帰）
- **リアルタイム性のあるゲーム**（アクション・シューティング・リズム）— 到達可能性テストは
  書けることもあるが、フレーム精度の入力・操作感・当たり判定・難易度は検証できずフレーク化する。
  最悪なのは「全緑＝安心」の感覚だけが持ち越されること：**ハーネスが守ってくれない領域で
  green を信じるのが一番危険**
- **乱数・プロシージャル生成** — 通しシナリオの決定性が崩れる。使うなら「シード注入」を
  セーブ注入と同じ「テスト入力の例外」として設計すること
- **ドラッグ／スワイプ／ピンチ主体の操作** — BaseDriver にはタップしかない。追加自体は容易だが、
  そのゲームの「動詞の文法」ごと設計してから足すこと
- **長編・大規模・マルチプレイ** — 「単一HTML・依存ゼロ・file://・localStorage」は
  プロトタイプの規約。規模が出たら卒業する設計

## 新しいゲームの始め方

1. このテンプレートから新リポジトリを作る（GitHub の「Use this template」か clone）
2. セッションは**新ゲームのリポジトリで**始める（テンプレート側からではなく）
3. 最初の指示は繋ぎ方ではなく**仕様の言語化**に使う。例：

   > html-game-template から立ち上げてある。仕様：〇〇なゲーム。コアの動詞は△△、
   > 進行の節目は□□、雰囲気は◇◇。CLAUDE.md の規約に従い、遊びを足すたびに
   > シナリオも足して green を維持して。

4. 最初のセッションでは、実装に入る前に **`/template-catchup`** を実行する——テンプレート
   最終更新からの空白期間分の世界のドリフト（Playwright・Node・Claude Code）をここで検出し、
   直した分は還流で正本に返す（立ち上げ日の不意打ち修理を、予定された一手目に変える工程）
5. CLAUDE.md 冒頭にゲーム名・一言仕様・「このゲームの構造」を書き、スターター
   （`index.html`）と `harness/driver.js`・`harness/scenarios/` をそのゲームのものに
   置き換えていく。`framework/` と `.claude/`・`KNOWLEDGE.md` の設計原則はそのまま持ち越す
6. 設計スパーリング（claude.ai 等）を経て立ち上げる場合は、仕様書と一緒に
   **「決めたこと・却下した案とその理由」を `DECISIONS.md` へ移す**——仕様1枚に圧縮すると
   「なぜ」が落ち、却下済みの案が後のセッションで再提案される。仕様書を改版したら
   旧版は `_archive/` へ（先頭に `superseded by 〈最新版〉` を書く。CLAUDE.md の鮮度ルール）

## 動作確認

```bash
(cd framework && npm i) && (cd harness && npm i)   # 初回のみ
node harness/run.js                                # 3プロファイル×通しシナリオが green のはず
```

スターターは `index.html` をブラウザで開くだけで動く（`?debug` でデバッグボタン）。

## 運用の約束（境界の還流）

`framework/` と `.claude/` は各ゲームに vendored コピーされる**写し**で、正本はこの
テンプレート。新しいゲームを作ると「これは基盤に要る／ゲーム固有だった」という境界の修正が
必ず出る。それは失敗ではなく工程。**基盤への直しはゲーム側リポジトリで済ませず、この
テンプレートに還流させる**（緩い結合：バージョン固定の依存より、テンプレート＋還流ループを
優先する）。

この還流はワークフローとして焼き込んである：テンプレートから作ったゲームには
`/framework-backflow` スキルと「framework/ を触ったら同じセッションで還流」という
CLAUDE.md の規約が複製されるので、Claude で開発している限り、境界修正は自動的に
正本へ戻ってくる（初回だけ、ゲーム側セッションにこのリポジトリへのアクセスを
許可する必要がある）。スターターの通しシナリオは BaseDriver API の互換性テストを
兼ねており、還流が既存 API を壊せばここで検出される。
