# ハーネス・ノウハウ台帳

このファイルは「見つけたバグのクラス → ハーネスへの恒久対策」の対応を蓄積する台帳。
**新種のバグを見つけたら必ずここに追記し、可能な限り driver の不変条件チェックか
シナリオのアサーションに落とすこと。台帳に書くだけで終わらせない。**

下の設計原則は前作群（録音プロトタイプ proto_ae などのタップアドベンチャー系）で
実際に踏んだバグ・プレイテスト指摘から抽出した持ち越し資産。**このゲームで
新しく学んだことは、原則なら「設計原則」へ、個別バグなら「バグ台帳」へ足す。**

> **ジャンルが違うゲームでの読み方**: 原則はタップ操作の2Dゲームの系譜で抽出されたもの。
> 字面のまま適用できない場合は捨てるのではなく、**同型の問いに翻訳して**適用する
> （例:「オブジェクトの座標を重ねない」→「入力ターゲットを曖昧にしない」。
> 「canvas に DOM を重ねない」→「遊び場の入力を UI が奪わない構造にする」）。

## 設計原則（前作群から持ち越し）

### コアファンタジーの原則（全原則に優先。どのゲームでも）

0. **コアファンタジー・ゲート（道具化テスト）** — ギミックを追加・変更する**設計段階**で必ず問う：
   **「この仕掛け、テーマを剥がした汎用の道具（トンカチ・鍵・接着剤）に差し替えても遊びが変わらないか？」**
   変わらないなら、コアファンタジーの動詞で言い換えてから実装する。
   - 前作の実例：音を録って奏でるゲームで、道具が機能的に「壊す・固める」の工具と等価になり、
     プロデューサーに「ガジェットがトンカチになっている」と指摘された。摩擦の指摘だけを食べて
     30回以上改善した結果、その全部が「工具としての完成度」に向かっていた
   - **テスターは摩擦（詰まる・分かりにくい）は報告できるが、ファンタジーの欠落は報告できない。**
     プレイテスト後は毎回、定点質問**「◯◯している感じがしたか？（道具を使っている感じではなく）」**
     を聞く（◯◯＝このゲームのコアファンタジー。CLAUDE.md 冒頭に書いた一言仕様がそれ）
   - **ハーネスの緑＝正しさではない。** 到達可能性・遮蔽・詰みは測れるが、ファンタジーは測れない。
     測れるものだけが最適化される——測れないものは検証ではなく設計工程で守る。
     数revごとに全ギミックへ道具化テストを実施して台帳に記録する（fantasy review）
   - **フィードバックの翻訳層**：指摘を字義通り実装する前に「コアファンタジーの語彙でこの問題を
     解けないか」を一度通す。納得感をテーマ外の理屈（物理の整合性など）で作り始めたら黄信号

41. **触感の要件は、可能な限り測定可能な帯に翻訳して下限を固定する（原則0の補完）** —
    「ゆらっと揺れる」のような触感はそのままでは測れず、測れないものは静かに消える
    （減衰係数を1つ「安全側」に振っただけでオーバーシュートが1.7%＝不可視になり、
    検証は全緑のまま核の触感が消えた）。バネなら減衰比 ζ の帯・オーバーシュート回数、
    追従なら遅れの ms 帯のように**物理量の帯**に翻訳し、下限をハーネスのアサーションに固定する。
    翻訳しきれない残り（質感・気持ちよさ）は原則0どおり設計工程と実機プレイで守る
    （出自: てんびん v0.1 バグ#3 — コア触感「1〜2回ゆらっ」を『(角度−目標)の符号反転回数≥2』に翻訳して恒久検出）

47. **仕様は実装前に、文脈を共有しない評価者の反証にかける（作成者と評価者を兼ねさせない。
    原則0の設計工程を守る装置）** — AIとの合意形成（ブレスト→仕様化）で作った仕様は
    内部一貫性が高いぶん、**最初の思い込みを全工程に引き継ぐ**。原則0は「測れないものは
    設計工程で守る」と言うが、道具化テストも含め設計工程の関門は全て作成者の自己点検で、
    仕様そのものを外から壊す工程が無かった。コードの独立監査（progress-auditor）と同じ分離を
    仕様にも適用する：**仕様書だけ**を渡した別文脈のエージェントに「30分プレイした実プレイヤーが
    離脱する理由を、操作・退屈・意味不明の3軸で挙げよ。設計意図の擁護は禁止」と壊させ、
    所見をトリアージ（仕様修正／TODO_SYNC の判定待ちへ／棄却＋理由）してから実装に入る。
    同じ文脈で自問しても反証にならない——評価者が作成者バイアスに感染していないことが要件
    （プロンプトに設計意図・経緯を書いた時点で失格）。所見は独立監査と同じく鵜呑みにしない。
    反証の単位は仕様の版（毎変更にかけると所見が反復ノイズ化してゲートが形骸化する）。
    運用は `/spec-falsify` スキルと CLAUDE.md「仕様反証ゲート」（反証スタンプ＋SessionStart hook）

### テストの原則（どのゲームでも）

1. **実入力主義** — テストは本物の pointer イベントのみでゲームを操作する。
   `evaluate` での状態書き込み・内部関数呼び出しは禁止（読み取りのみ可）。
   内部関数を呼ぶテストは「全行程green」でも操作可能性を何も証明しない
   （前作の実例: 同座標オブジェクトのタップ横取りによる進行不能を長期間素通しした）。
   例外は **localStorage へのセーブ注入**のみ（セーブ/ロード試験の「入力データ」であり、
   ゲーム内部状態の直接書き換えとは区別する）。乱数を使うゲームでは**シード注入**も
   同じ「テスト入力の例外」として設計してよい
2. **通しシナリオ＝到達可能性の証明** — 進行フラグを追加・変更したら、
   そのイベントに実操作で到達するステップを通しシナリオに追加する
3. **モード安定待ち** — 遅延発火の演出・ダイアログがあるゲームでは、
   「操作可能モードが一定時間続く」まで待ってから次の操作をする（settle 待ち。
   タップアドベンチャー系での実装例が pumpUntilPlay）
4. **3プロファイル必須** — desktop / mobile縦(390x844) / mobile横(844x390) を全て回す。
   DOM UI は CSS px 固定なので、はみ出し・遮蔽は該当画面サイズでしか検出できない
   （前作の実例: 縦持ちだけ直して横持ちの遮蔽を見逃した）
5. **到達可能性≠発見可能性** — 正解を知っているシナリオは「答えが分かるか」を検証しない。
   新パズル追加時はスマホ実寸のスクリーンショットを撮って判読性を目視確認する

39. **実時間依存のゲームは、固定タイムステップ＋timescale 注入で「加速可能」に設計する** —
    ゲームループを固定 Δt（例 1/120s）のアキュムレータ方式にし、`?timescale=N` は
    1実フレームあたりの積分回数を増やすだけにする（**Δt 自体は変えない**——バネ等の数値安定性を守る）。
    静止判定・遷移・長押し・ログの ms 計測などゲーム内の全タイマーはシミュレーション時間基準に
    統一する。これで「静止1秒で完了」のような実時間仕様を含む通しシナリオが実時間の数分の一で回る。
    原則1の注入例外（セーブ・シード）に並ぶ第3の注入として、テストフック契約に明記する
    （出自: てんびん v0.1 — 5問セッション通し（静止判定1s×5＋演出）を timescale=8 で約90秒に収めた）

40. **「出さない」と約束したもの（禁止演出・禁止API）は、ラップで機械検証する** —
    「✗を出さない」「文字を描かない」のような**負の要件**は、目視でもシナリオの成功でも
    回帰を検出できない。ハーネスが描画・音声 API のプロトタイプ
    （例 `CanvasRenderingContext2D.prototype.fillText`）を注入スクリプトでラップし、
    許可された文脈（例: 親メニュー表示中）以外の呼び出しを違反として数えれば、
    **全テストシナリオが同時に禁止事項の監視網**になる
    （出自: てんびん v0.1 — 「文字・数字は親メニュー以外で全面禁止」を fillText/strokeText ラップで常時監視）

### 進行・セーブの原則（セーブを持つゲームなら何でも）

6. **進行フラグは「操作の瞬間に確定」か「起動時に逆算修復」** — 遅延タイマーや演出の
   コールバック内で進行フラグを確定させない。やむを得ない場合は `reconcileSave()` に
   逆算修復を必ず追加し、セーブ復旧シナリオにケースを足す
   （前作の実例: フラグ確定が録音演出のコールバック内にあり、詰む中間セーブ窓が5箇所あった）

44. **セーブの破損耐性は、容器だけでなく要素レベルまで検証してから使う（同族: #6）** —
    「配列であること」の検証だけでは `[null]` が素通りし、要素をオブジェクト前提で読む画面
    （統計・一覧）を開いた瞬間に落ちる。起動は通るぶん発症が遅れ、破損データ起因なのに
    「メニューを開くと固まる」という別種のバグに見えて診断が迷走する。読み込み時に要素を
    フィルタ（サニタイズ）し、以後の書き戻しでストレージ自体が浄化されるようにする。
    ハーネスには「要素レベルで壊れたセーブの注入 → 全画面を開く」ケースを置く
    （出自: てんびん v0.1 バグ#6 — `tenbin_logs=[null]` で親メニューを開くと TypeError → 全フリーズ）

### UI 構造の原則（2D・canvas 系。別構成なら同型の問いに翻訳）

7. **インタラクティブ DOM は canvas に一切重ねない** — 遊び場のタップ対象を DOM が奪う
   事故を構造で断つ。UI は上部バー＋下部ドックに分離し、fitCanvas が両バー分を差し引く。
   バーの高さは固定し（内容の変化でレイアウトを動かさない）、`#wrap` の変化には
   ResizeObserver で fitCanvas を追従させる（前作の実例: 道具解放でドックが高くなり
   canvas に重なった）。invariant「バーは canvas と重ならない」で恒久検出
   （→ BaseDriver.checkBarsClearOfCanvas）
8. **入力ターゲットの判定を重ねない** — タップ判定は最近傍優先でも、同座標は常に曖昧。
   invariant「同座標(d<12)で fail・近接で warn」で恒久検出（→ checkTapTargetSeparation）

### 遊びのデザイン原則（UX。プレイテスト指摘の再発防止。ジャンルに応じて翻訳して適用）

番号は歴史順の恒久ID。ここではクラスタ順に並べる（索引表と同じ区分）。

#### 情報開示（何を語り、何を黙るか）

9. **ヒントの3層** — ①世界の表示が状態を語る（アイコン・脈動・プレビュー）
   ②調べた文は観察の独白（手順・道具名を言わない） ③手順は案内役NPCに聞いた時だけ。
   フィールドのテキストに「（〜すれば……）」の指示口調を書いたら負け。
   見れば分かる状態（開/閉など）をテキストで重ねない。進捗カウンタも書かない
12. **ゴールの予告を環境で見せる** — 仕掛けが完成したら何が起きるかを、未完成の状態が
    示唆する（例: 風がつかえて軋んでいる＝通せば鳴る）。1エリア＝1目的で焦点を絞る
18. **状態表示は「現在」だけを語る** — マークやプレビューを**未来の結果**（分岐の行き先に
    「これが正解」の印など）に貼ると、パズルの答えの直渡しになる。現在の状態（今どう鳴って
    いる・今どこで止まっている）だけを表示し、結果は実行して初めて分かるようにする

49. **謎解きの正解は「プレイヤーが照合で推理できる関係」に置き、照合の両側を提示する（色鍵テスト）** —
    正解を鍵にするなら必ず問う：**「この対応、無地の色付きの鍵と扉に差し替えても遊びが変わらないか？」**
    変わらない（＝覚えゲー・当てずっぽう）なら、性質の一致・律動の一致・類推・部分反応の勾配など、
    観察が推理になる関係へ置き換える。さらに関係だけでは足りない——**お題と候補の両側を、
    プレイヤーが同じ手段でその場で確認・比較できるようにする**（片側が文字説明だけだと
    照合の瞬間＝アハが起きない）。誤答には必ず部分反応を返す（総当たり化の防止）。
    **世界のルールは冒頭に宣言し、以後の全ての因果をそのルールで説明する**——宣言されていない
    ルールで解ける謎は「なんでこれで正解なのか分からない」を生む
    （出自: proto_ae v4〜v4.3 — プレイテスト2回連続「当てずっぽうでアハがない」を、
    お題の可聴化＋A/B聴き比べ＋世界ルール2行宣言で解消した）

50. **単一の感覚チャネルだけが運ぶ情報を作らない** — 音・色・触覚など、環境や身体条件で
    失われうるチャネル（ミュート再生・色覚多様性・振動オフ）だけに判定材料を載せない。
    全ての判定材料に**別チャネルの翻訳**を与える。翻訳は恣意的なラベル（文字・記号の対応表）
    ではなく**物理の写し**（高さ＝密度、律動＝点列、質感＝粒/面）にする——ラベルは暗記に
    なるが、写しは元チャネルと同じ推理がもう一方でも成立する。翻訳は**単一のレンダラ**で
    全表示面に描き（表示先ごとに別実装を作らない）、お題と候補の両側に同じ翻訳を出す。
    「全対象が翻訳を持つ」「お題と正解の翻訳の型が一致する」はハーネスで恒久検査する
    （出自: proto_ae v4.4 — プロデューサー制約「音を消してプレイする人にも当然伝わること」。
    全SEに音紋 print を義務化し、対応表の型一致を invariant に落とした）

#### 教習と動詞（操作の学ばせ方）

10. **動詞の文法を最初に固定し、全編で通用させる** — 「タップ＝調べる」「ボタンA＝◯◯」の
    対応を決めたら例外を作らない（最低でも無反応にしない）。教えた動詞が特定の場所だけ
    効かないと、プレイヤーは詰みと区別できない
13. **道具の教習は「気づく→自分で押す→説明」の順**（同族: #15） — 解放時の自動デモは受け身
    （前作で一度採用し、後に巻き戻した）。①解放は「押せるようになった」の告知＋ボタンの
    脈動に留める（大きな解放は、対象ボタン以外をグレーアウトするスポットライトで効果を
    説明してよい。矢印はボタンの実座標に合わせる） ②初押しの瞬間に一言で意味を語る
    （対象を壊した時は効果のトーストが上書きしてよい——効果自体が説明になる）
    ③空撃ちが玩具になる ④ボタン表記は効果ファースト ⑤初使用まで新ボタンは脈動して
    「押してみて」と誘う
15. **教習ゲートは「次へ進む扉」に置く**（同族: #13） — 破った結果が過去の便利化ではなく
    前進になるように。恒久化できる仕掛けは恒久化する（毎回のやり直しは作業）
19. **能力の制約は操作の制約として体感させ、ルールは1本に固定する** — 「一定時間だけ効く」力
    なら、ボタン側にもゲージ（押せる時間＋クールタイム）を付けてルールを一致させる。
    能力が世界に及ぼす作用は1本の理で説明できる形に固定し、場所ごとに理屈を発明しない

#### 即時フィードバック（1手ごとの返事。「受理」は降ろし節の #33 が同族）

11. **状態トグルには即時の結果プレビューを必ず付ける**（同族: #20・#33） — 総当たりでしか
    解けない配置は解けても「アハ」にならない。1手目の結果が次の1手を示唆する初期状態を選ぶ
14. **押している間だけ起こる操作は、リアルタイムに反応させる** — 長押し系は押下と効果を
    同期させる（固定長で再生すると操作と結果が乖離する）
20. **トグル系パズルは1手ごとに短い結果演出を返す**（同族: #11・#33） — 一手打つたび
    「どこまで通ったか」を見せ聴かせ、演出中は次の入力を受けない。連打でも因果が読め、
    偶然の正解でも「なぜ解けたか」が分かる

#### 演出の規律

16. **最大の演出打点は、見せるのではなく操作させる** — クライマックスはプレイヤー自身の
    入力で成就させる（自動再生は受け身）
17. **状態遷移の因果は1本の演出チェーンで見せる** — 大きな変化を「裏の数値」で起こさない。
    原因→変換→結果が絵と音で追えること。**納得感＝因果が絵と音で追えること**
21. **演出は直列のキューで——同時発火で重ねない** — フルスクリーン演出（回想等）と告知
    （解放トースト等）を同時に出すと後者は読めずスキップ扱いになる。**進行フラグの保存は
    即時**（演出中に離脱しても失われない）、**提示は前の演出が終わってから**。
    メッセージが「止まった」と言う瞬間に、絵も本当に止める（言葉と状態の同時性）

#### 遅延・入力の安全

22. **遅延実行する演出は、発火時に前提を再検証する** — setTimeout で
    遅らせた演出・環境音・トーストは、発火時に部屋／モード／状態が前提のままかをガードして
    から実行する（プレイヤーは待ってくれない——その間に部屋を移動する）。部屋の環境音のような
    状態依存の決定は1関数に一元化し、入室時とセーブ復帰の双方で同じ関数を使う
23. **永続メッセージは実体を持たせる** — メッセージを「タップで閉じる」方式にするなら、
    メッセージ自体に pointer-events を与え、**閉じるためのタップが真下の盤面へ透過しない**
    ようにする。ハーネスの tap も「目標座標がメッセージに覆われていたら、閉じてから再タップ」
    の実プレイ準拠にする（前作の実例: 入室メッセージの真下がパズルの操作対象で、
    閉じるつもりのタップが盤面を変えていた）

42. **モード外の入力は既定で遮断し、演出キューは消費時に対象の前提を再検証する（同族: #22）** —
    演出・遷移・リザルト中もフィールドの入力を素通しすると、演出キューが開始時に写し取った
    対象（食べる予定のオブジェクト等）をプレイヤーが同時に動かせてしまい、消費時に前提が
    壊れてクラッシュ級になる。①入力ハンドラは「このモードで許可する動詞」を明示して既定拒否、
    ②キューは id 参照にして消費の瞬間に対象の状態を再検証（#22 の演出キュー版）、
    ③解除系の操作（載せたものを外す等）は二重解除に耐える、の3層で守る。
    ハーネスには「演出の最中に対象へ介入する」実操作ケースを置く
    （出自: てんびん v0.1 バグ#4 — 完了演出中に皿のりんごをつまむと食べキューが TypeError → rAF 恒久停止＝進行不能）

43. **毎フレーム導出される状態を入力判定・公開読み取り面が参照するなら、初期化・リセットの
    瞬間にも確定する（同族: #6）** — 「どうせ次のフレームで計算される」座標・レイアウトは、
    init〜最初のステップの窓では未初期化値のまま入力判定とテストフックに晒される。
    人間には踏めない一瞬に見えても、E2E（ロード直後に操作）は確実に踏む——しかも症状は
    「たまに変な場所に落ちる」というフレークに見えて、原因追跡が長引く。導出状態の計算関数を
    初期化・リセット処理からも同期的に呼び、ハーネスは「公開直後の最初のスナップショットが
    有効値」をアサートする
    （出自: てんびん v0.1 バグ#1 — 皿座標が最初の物理ステップまで (0,0) で、起動直後のドロップが皿外判定になった）

### 音の原則（WebAudio。ブラウザ/モバイル実機。時間解像度 v0.2a の「4層の無音」から確立）

24. **音の開始は「無音のタイトルへの一触れ」に乗せる** — 自動再生ポリシー上、初回の
    ユーザージェスチャ前に音は出せない（どのブラウザでも）。無音のタイトル画面を挟み、
    その一触れで解錠して、タイトルの溶暗と音の立ち上がり（アンカー付きランプ）を重ねる——
    制約が演出になる。既定は「鳴る」にし、ボタンは消音トグルにする（音が既定オフだと
    小さなボタンの発見が前提になり、実機で「音が無い」と判定される）
25. **iOS の解錠は touchend/click で、走るまで再試行する** — iOS WebKit は
    pointerdown(=touchstart) を音の解錠ジェスチャとして認めないことがある。
    pointerup/touchend/click で `AudioContext.resume()` と無音 `<audio playsinline>` の再生
    （本体の消音スイッチ対策）を、`state==='running'` になるまで毎回再試行する。
    音トグルの表示は実状態（鳴＝走っている/待＝解錠待ち/消）を反映させる——
    実機で無音のとき、どの層で詰まっているかをユーザーの目視だけで診断できる
26. **AudioParam のランプは必ず setValueAtTime でアンカーしてから予約する** — WebKit は
    アンカーの無い linearRamp / exponentialRamp を発火させず、gain が初期値のまま
    永久に無音になる。Chromium は現在値から補間するため、手元の検証は緑のまま素通しする。
    `cancelScheduledValues(t) → setValueAtTime(現在値, t) → ramp` を定型にする
27. **音の検証は「実信号」と「可聴帯域」で測る** — `AudioContext.state` では「鳴っているか」を
    測れない。AnalyserNode を master に挟み、①実信号 RMS > 閾値 ②可聴帯域（250Hz〜6kHz）に
    エネルギーがある、をハーネスで機械検査する。エネルギーが基音200Hz未満に集中すると、
    信号があってもラップトップ/スマホのスピーカーでは物理的に聞こえない
    （音圧はコンプレッサー＋可聴レジスタで作る）。ハーネスのブラウザが Chromium のみなら、
    **音に触れる変更は iPhone 実機確認を必須**にする（25・26 は Chromium で再現しない）

### シミュレーション・実装・ハーネスの原則（ゲームに限らず開発一般に通じるもの）

（旧・開発一般テンプレートから降ろしていた原則群。テンプレート統廃合で**本台帳が唯一の
正本**になった——`DECISIONS.md` D1。番号は恒久IDのまま維持する）

28. **不変条件は共有ヘルパに集約して全対象へ適用する** — 「〜禁止」「全対象に〜」型の
    不変条件を代表例（目立つ一箇所）にだけ実装すると、他の対象が黙って違反したまま残る。
    不変条件は1つの関数・1つの経路に集約し、ハーネスは対象を列挙して対象ごとに検査する
    （出自: 時間解像度 v0.2 — 「完全静止禁止」を雨粒にだけ実装し、他の存在の凍結が監査まで残存した）
29. **追従系の操作ドライバは「目標」と「状態」の両収束を待つ** — 入力がなめらかな追従
    （グライド・イージング・慣性・デバウンス）を挟む操作では、ドライバが追従状態だけ
    見て操作を止めると、入力目標のオーバーシュートで測定中に状態が流れ続ける。
    入力目標値と追従状態の双方が目標に収束してから測定に入る
    （出自: 時間解像度 v0.2 — ダイヤル追従値だけ見て止めたハーネスの測定値が別帯域相当にずれた）
30. **打ち消し合う力には進行保証を入れる（ライブロック防止）** — 進行を打ち消す効果
    （押し戻し・減速・リトライ抑制等）が複数ソースから**加算**される構造では、合算が
    どの終了条件も発火させない均衡（ライブロック）を作れてしまう。合計の上限クランプ・
    純進行の下限・保険のタイムアウトのいずれかで「必ずいずれかの終了条件に到達する」ことを
    構造で保証し、**重ね掛け（複数ソース同時）の検証ケース**を足す——単発ソースのテストでは
    決して現れない
    （出自: プリズマの編み手 v0.1 — 領域効果を3つ重ねると敵が前進も敗北もできず戦闘が永遠に終わらなかった）
31. **飽和はその内側の補正を無音で消す（クランプと補正の適用順序）** — 上限クランプ・飽和・
    丸めの**内側**（前段）で適用された補正・効果は、飽和域では出力に一切現れない——
    「効果を約束する表示」と「変わらない実挙動」の乖離が静かに生まれる。約束した補正は
    飽和の**外側**（後段）で適用し、クランプを導入したら「クランプ域で全ての約束済み補正が
    観測可能か」を検証観点に足す（原則30の対策が生んだ欠陥クラス——対策が次の欠陥を生む連鎖の実例）
    （出自: プリズマの編み手 v0.1.1 — クランプの内側で学習効果を適用し、表示だけ変わって挙動が数理的に不変だった）
32. **時間積分型の効果への乗算耐性は、曝露時間が支配する** — 「滞在時間×レート」で効く効果
    （DPS領域・継続コスト・累積ペナルティ）への百分率の耐性・補正は、積分時間が十分長いと
    結果を何も変えない。結果を本当に変えたいなら積分の**上流**（曝露時間・対象の挙動・
    レート上限）を変えるか、判定閾値を下回るまで**振り切る**。成立帯は数理で導出してから
    採用し、境界値は invariant としてコード内コメントに残す
    （出自: プリズマの編み手 v0.1.2 — 「耐性70%減」が支配戦略を全く止められず、導出すると×0.03まで振り切る以外に解が無かった）
33. **操作・支払いの受理は、その瞬間・その場所に可視化する** — 応答の本体はゲーム内時間・
    演出の都合で遅れてよいが、**受理の確認はプレイヤーの主観時間で即時**に、操作した場所に
    出す。原則11・14・20 が「結果」の見せ方を扱うのに対し、これは「受理」（＝自分がやった、
    の帰属）を扱う——受理が見えない操作は帰属不明になり、コストを支払わせる設計
    （待つ・滞在する・溜める）は受理が見えないと支払いが続かない
    （出自: 時間解像度 v0.2a — 雨粒タップの帰属不明・樹への滞在の徒労感を、接触点に滲む受理サイン〔墨の輪＋雫の音〕で解決）
34. **幾何の被覆問題を、量の閾値で解かない** — 「届く／覆う／漏れる」が距離・範囲の不等式で
    決まる系（攻撃半径・当たり判定・射程）で望む挙動が出ないとき、HP・回数・タイミングなど
    **量の閾値**で補修すると、離散周期との位相依存（かすめ通過が周期より短いとくじ引きになる）や
    別の不変条件の破壊を生む。被覆の不等式を書き下し→**成立帯の存在を確認**→帯の中央に
    パラメータを置き→境界を実測して invariant コメントに固定→「帯から出たら赤くなる」
    アサーションを残し、検出器が本当に検出することを一度実証する
    （出自: プリズマの編み手 v0.1.3 — 攻撃領域の半径が防衛対象の到達円を完全被覆して
    「誘導で守る」遊びが幾何的に不能なのを敵HP3倍で補修し、攻撃tickと通過時間の位相くじ＋
    進行段階の強さ逆転を併発した）

45. **メインループ（rAF）には最終防衛線を張る——ただし検証には見える形で** — step/render の
    例外が1つ漏れると `requestAnimationFrame` の再予約に到達せず、描画・入力・物理の全てが
    恒久停止する（プレイヤーにはリロード以外の復帰手段が無い＝進行不能の最悪形）。ループ側で
    catch して再予約は必ず実行し、握りつぶす代わりに `console.error` へ出す——ハーネスが
    console.error / pageerror を fail として収集していれば、**実機では止まらず・検証では隠れない**
    が両立する。防衛線はあくまで最後の層であり、既知の例外経路は個別に潰す（#42・#43・#44）
    （出自: てんびん v0.1 — 独立監査が「例外1つでループ全死」の構造を指摘。frame の try/catch＋
    ハーネスの console.error 収集で二重化）

46. **「トレランスに入れて保持する」操作には、入力の最小粒度がトレランス幅を下回る操作特性を
    与える（同族: #29・#41）** — 「±N°に合わせる」「帯に収めて維持する」型の操作は、
    入力の最小ステップ（イベント往復のオーバーヘッド × 等速レート × timescale）が
    トレランス幅を超えた瞬間、人間もテストドライバも目標を跨いで永久振動する。
    しかもたまたま着地した回は緑になるので、**フレークとして現れて設計欠陥に見えない**。
    押し始め低速のイーズイン・可変レート等で最小粒度＜トレランスを構造で保証し、
    ハーネスの閉ループ操作（実入力での収束）自体を検証器として維持する。
    timescale 加速は最小粒度を N 倍に太らせることに注意（実時間 10ms の押下が
    シミュ時間 10N ms になる）
    （出自: 正位置 v0.1 バグ#2 — 一人称の視線±2°収束が等速90°/s・timescale4 で最小ステップ8〜12°になり永久振動）

48. **検証環境（モック）の自由度は、本番の制約に合わせて縛る（有限アセットバンク方式）** —
    プロトタイプ・モックが本番で作れないもの（動的生成の音・無限のアセット・本番に無いAPI）を
    作れてしまうと、**その自由度の上に設計を建ててしまい**、検証が緑のまま実装不能な仕様が育つ
    （検証環境の勾配が本番と別方向を向く）。使ってよいアセット・能力は **ID付きの有限バンク**
    （＝そのまま発注リスト）に列挙し、モックは各IDの仮実装としてのみ許す。新しいアセットを
    使いたければ必ずバンクに行を足す——バンクが常に真実であり続ける発注リストになる
    （出自: proto_ae rev.23 — 実行時の音高導出の上に音楽ギミックを建てたが、実機は固定サウンド
    再生のみで棄却。仮想サウンドバンク `BANK` として恒久化した）

### 読み合い・対抗のデザイン原則（プリズマの編み手 v0.1 系より。ジャンルに応じて翻訳して適用）

35. **読み合い情報は「内部キー⇔表示文言⇔実効果」の三点一致、フレーバーの含意まで** —
    敵の学習・耐性・カウンターをプレイヤーに明示するゲームでは、表示と実挙動の不一致は
    進行不能と同格のクリティカル欠陥として扱う（偽情報の上の推理は成立しない）。
    対応表を契約（テスト側との共有ドキュメント）に固定し、キー⇔文言はシナリオで機械検証する。
    **フレーバー文が実効果を超えた対抗を語る**のも同罪（「毒を刺す前に逃げるぞ」と書いて
    毒の成立率が1ミリも下がらない等）。効果がクランプ・飽和の内側に置かれて
    実挙動に現れないケース（原則31）も含めて、学習を足すたび三点を同時に更新する
    （出自: プリズマの編み手 v0.1.1〜v0.1.3 のバグ台帳 #4・#5・#9——3度踏んだ）
36. **支配戦略はテストで禁止する** — 単色・単手のスパムで通しをクリアできる構成が
    一つでも成立すると、他の全ての手が「お飾り」になり読み合いが死ぬ（プレイヤーは
    「他のを置く理由がなかった」と正しく報告してくる）。①コスト・資源制約で枚数を
    構造的に縛り、②カウンターは対象戦術を**実際に脅かす強度**に置き（成立帯を数理で
    導出してから採用——中庸な値の成立帯が存在しないことがある。原則32・34）、
    ③「スパム構成が最初は勝てるがカウンター後に必ず敗北する」通しシナリオで恒久検出する。
    可読性を壊す変数（敵速度の不連続・視認不能な高速化）をバランスのレバーにしない
    （出自: プリズマの編み手 v0.1.2 — 火のみ×5が全シードで完全勝利し、学習も2戦目以降沈黙していた）
37. **解禁は「完結したシナジーの授与」の単位で渡す** — 新要素の解禁は、単体で意味が
    読める（誘導の柵）か、対になる相方と同時（濡らす水＋落とす雷）にする。相方が
    1段階遅れて届く解禁は「これは何のためにあるのか」になる（プレイヤーは
    「シナジーが無いのに最初から解禁されているのはなぜ」と報告してくる）。
    解禁の瞬間にシナジーを一言で教示する画面を挟む（推理の材料の渡し方そのもの）
    （出自: プリズマの編み手 v0.1.2→v0.1.3 — 水を先行解禁して雷を後にした構成が不評で、対渡しに再編）
38. **経路の誘導は、斥力の横成分で設計する** — 敵・対象を「避けさせて導く」仕掛けは、
    対象の進行軸に対して**斥力の横成分が残る配置**だけが経路を曲げられる。進行軸を挟んで
    対称に置いた斥力は相殺し、ブレーキにしかならない（実測で bare と完全同値になる）。
    誘導の実効は「柵あり／なしを同一シードで比較して被害が減る」シナリオで恒久検出する
    （出自: プリズマの編み手 v0.1.3 — 横並びの風柵が敵を1体も曲げられず、進入軸上の縦2連で初めて誘導が成立）

## 追記ルール

- 1バグ = 1行。**症状 → 根本原因 → 対策** をこの順で。対策は「driver/シナリオのどこに実装したか」まで書く
- ゲーム側の修正だけで済ませず、「同じクラスのバグが再発したらハーネスが検出できるか？」を自問する。
  できないなら invariant を足す
- 原則として一般化できる学びは「設計原則」の節にも番号を振って足す（→ 足したら `/framework-backflow` で正本へ）
- **肥大化したらアーカイブ**（proto_ae の台帳運用から）: バグ台帳が30行を超えたら（または全体が
  300行を超えたら）、対策がコード化済み（invariant／シナリオに実装済み）の古い行から順に
  `harness/KNOWLEDGE-archive.md` へ移す。本体には「#番号｜一行要約」の索引を残す。
  設計原則は生きた知識なので、原則はアーカイブせず本体に残し続ける

## サブエージェント運用の雛形

進行フラグ・入力処理・インタラクション対象の配置に触れた変更では、実装とは別の文脈で監査を回す：

- **進行不能監査**（progress-auditor）: 「各進行イベントについて到達経路を逆向きに辿り、
  必要なフラグ・オブジェクト・道具がその時点で成立しうるか検証。モードスタック経路、
  setTimeout×画面遷移の競合、旧セーブ再開の詰み、入力ターゲットの重なりを重点確認。コード引用で根拠を示す。編集禁止」
- **ハーネス実行**（harness-runner）: `node harness/run.js` を実行し、report.json の fail/warn を報告させる
- 監査所見は鵜呑みにせず、ハーネスのシナリオで再現できたものだけをバグとして扱う
  （再現できたら台帳へ、シナリオ/invariant へ反映）

## 既知の制約

- **緑＝正しさではない。** ハーネスは到達可能性・遮蔽・詰みを測るが、コアファンタジーは
  測れない。測れないものは検証ではなく設計工程（原則0）で守る。数revごとに全ギミックへ
  道具化テストを実施し、結果を台帳に記録する（fantasy review）
- リアルタイム性（フレーム精度の入力・操作感・当たり判定・難易度）は検証できない。
  緑を過信しない（README「適用範囲」参照）
- 音（Web Audio）の「正しさ」は自動検証していない。無音クラッシュ・JSエラーのみ検出。
  音楽的な確認は手動プレイで行う
- ブラウザは /opt/pw-browsers/chromium を前提（PLAYWRIGHT_CHROMIUM_PATH で上書き可）
